# Master Document — Performance Measurement & Conversion Loop

**Project:** Contextual Ad Intelligence Engine (Bangladesh-focused)
**Stage:** Hackathon MVP (1–3 day build)
**Scope of this document:** Performance tracking layer — how the system measures whether ads worked, what data gets stored, and what the brand-facing dashboard shows.

---

## 1. Purpose

This layer answers one question: **"Did the user take the offer?"**

Every ad generated by the system is shown to a persona (real or simulated). The persona either *converts* (takes the offer) or *doesn't*. That binary signal is the only metric we track. From thousands of these yes/no records, the system learns which creative formats work best for which audience segments.

This data feeds the Optimization Layer (Document 4), which uses it to bias future generation toward winning patterns.

---

## 2. Design Principle — Binary Conversion Only

We deliberately reject vanity metrics (CTR, view time, engagement scores). The only signal that matters is:

> **Did the user take the offer? Yes or No.**

### Why this matters

- **Honest:** No fake numbers, no inflated engagement, no questionable attribution
- **Defensible:** Judges cannot poke holes in a binary outcome
- **Simulatable:** Easy to model with demo personas
- **Actionable:** Direct mapping to "do more of what wins, less of what loses"
- **Production-ready:** Real brand integration plugs in real conversions (purchase, signup, redemption) without changing the system

### Pitch line to memorize

> *"We track one thing: did the offer get taken. Then we learn which formats and angles work for which people, and we generate more of what works. No vanity metrics. No fake attribution. Just outcomes."*

## 4. Format Taxonomy (Critical — Tag Every Ad)

Every ad must be tagged with its creative format(s) at generation time. The dashboard slices conversion data by these tags. Without consistent tagging, the dashboard cannot work.

### 4.1 Format categories

**Visual format** (one per ad):
- `static_image`
- `gif_template` — template-based GIF with swap-in
- `gif_ai_multiframe` — AI-generated multi-frame GIF
- `animated_banner` — CSS/JS animation
- `video_mp4` — template MP4
- `voice_audio` — TTS audio only

**Copy style** (one or more per ad):
- `joke_pun` — humorous one-liner
- `cultural_reference` — references local event/moment
- `question_hook` — opens with a question
- `stat_hook` — opens with a number/statistic
- `inspirational_quote` — aspirational, quote-style
- `urgency_driven` — time-limited framing
- `direct_offer` — straightforward deal mention
- `story_narrative` — short story-style
- `relatable_observation` — "we've all been there" style

**Tone** (one per ad):
- `warm_casual`
- `playful_energetic`
- `serious_premium`
- `urgent_action`
- `nostalgic_emotional`
- `informative_helpful`

### 4.2 Implementation

The LLM (Section 4 of Document 2) returns these tags as part of its structured JSON output:

```json
{
  "headline": "...",
  "body_copy": "...",
  "format_tags": {
    "visual": "gif_template",
    "copy_styles": ["joke_pun", "cultural_reference"],
    "tone": "playful_energetic"
  }
}
```

These tags flow through to the event logger.

### 4.3 File location

`/data/format_taxonomy.json` — Single source of truth. The LLM prompt references this file so it picks from the valid list.

### 5.2 Field meaning

| Field | Meaning |
|---|---|
| `converted` | Boolean — did the user take the offer? |
| `conversion_source` | `"simulated"` (MVP demo personas), `"live_interaction"` (judge clicked during demo), `"brand_integration"` (production with real brand data) |
| `context_snapshot` | Frozen context at moment of display — never derived later |
| `persona_snapshot` | Frozen persona profile at moment of display |
| `ad_tags` | The format taxonomy tags from generation |

### 5.3 Storage

For MVP, use a simple JSON file or SQLite. No need for a real database.

**File location:** `/data/events.json` (or `/data/events.db` for SQLite)

**Schema for SQLite (recommended):**

```sql
CREATE TABLE conversion_events (
  event_id TEXT PRIMARY KEY,
  timestamp TEXT NOT NULL,
  ad_id TEXT NOT NULL,
  brand_id TEXT NOT NULL,
  persona_id TEXT NOT NULL,
  context_snapshot TEXT NOT NULL,  -- JSON blob
  persona_snapshot TEXT NOT NULL,  -- JSON blob
  ad_tags TEXT NOT NULL,           -- JSON blob
  converted INTEGER NOT NULL,      -- 0 or 1
  conversion_source TEXT NOT NULL
);

CREATE INDEX idx_brand ON conversion_events(brand_id);
CREATE INDEX idx_persona ON conversion_events(persona_id);
CREATE INDEX idx_converted ON conversion_events(converted);
```

SQLite is preferred because the dashboard queries are simpler with SQL than with JSON filtering.

## 6. Conversion Decision Layer

This is where the yes/no signal comes from. Three modes, used in different situations.

### 6.1 Mode A — Simulated (MVP primary)

For pre-generated ads shown against demo personas, simulate whether each persona would convert. This makes the demo data look real without requiring live users.

**Logic:** Each persona has hidden "preference weights" for each format and context combination. The simulator computes a probability score and rolls a weighted random outcome.

**File location:** `/services/conversion_simulator.py` (or `.js`)

**Function signature:**

```python
def simulate_conversion(persona: dict, ad: dict, context: dict) -> bool:
    """
    Returns True (converted) or False (didn't) based on simulated 
    persona-ad fit.
    
    The fit score is computed from:
    - Persona format preference (some users prefer GIFs, some prefer text)
    - Context-ad alignment (hot afternoon + cooling product = high fit)
    - Tone-mood match (playful tone + relaxed mood = high fit)
    - Cultural relevance bonus (Pohela Boishakh ad during BD New Year = bonus)
    
    Score is converted to probability, then a weighted random roll decides.
    """
```

**Implementation sketch:**

```python
def simulate_conversion(persona, ad, context):
    score = 0.4  # baseline conversion probability
    
    # Format preference
    format_pref = persona["format_preferences"].get(ad["ad_tags"]["visual"], 0.5)
    score += (format_pref - 0.5) * 0.3
    
    # Copy style preference
    for style in ad["ad_tags"]["copy_styles"]:
        style_pref = persona["copy_preferences"].get(style, 0.5)
        score += (style_pref - 0.5) * 0.15
    
    # Context-product alignment
    if context_aligns_with_product(context, ad):
        score += 0.2
    
    # Cultural relevance bonus
    if context.get("cultural_moment") and references_moment(ad, context):
        score += 0.15
    
    # Clamp and roll
    score = max(0.05, min(0.95, score))
    return random.random() < score
```

**Why this works:** The probabilities aren't arbitrary — they're driven by realistic patterns (young users prefer GIFs more than older users, etc.). This produces dashboard data that *tells a story*, not random noise.

### 6.2 Mode B — Live judge interaction (demo wow moment)

During the live demo, let the judge be the user. Show them an ad. Give them two buttons: **"Take the offer"** or **"Skip."** Log their choice.

This produces a small amount of real interaction data and demonstrates the loop is real.

**File location:** `/frontend/components/JudgeInteractionPanel.jsx`

**Trigger:** Optional during demo. If time permits, run this for 2–3 ads to make the data feel "alive."

### 6.3 Mode C — Brand integration (production roadmap)

In production, the brand sends real conversion signals via a webhook:

```
POST /api/conversion-event
{
  "ad_id": "ad_xxx",
  "user_id": "brand_user_yyy",
  "converted": true,
  "conversion_type": "purchase" | "redemption" | "signup" | "install"
}
```

Mention this in the pitch. Don't build it. The interface is ready; the brand connects whenever they integrate.

### 8.3 Backend functions

**Function:** `get_format_winners(brand_id, date_from, date_to)`

```python
def get_format_winners(brand_id: str, date_from: str, date_to: str) -> list:
    """
    Returns conversion rate by visual format for a brand.
    
    Output:
    [
      {"format": "gif_template", "shown": 120, "converted": 50, "rate": 0.417},
      {"format": "static_image", "shown": 95, "converted": 27, "rate": 0.284},
      ...
    ]
    """
```

**SQL:**

```sql
SELECT 
  JSON_EXTRACT(ad_tags, '$.visual') as format,
  COUNT(*) as shown,
  SUM(converted) as converted,
  ROUND(SUM(converted) * 1.0 / COUNT(*), 3) as rate
FROM conversion_events
WHERE brand_id = ? AND timestamp BETWEEN ? AND ?
GROUP BY format
ORDER BY rate DESC;
```

**Function:** `get_segment_matrix(brand_id, date_from, date_to)`

```python
def get_segment_matrix(brand_id: str, date_from: str, date_to: str) -> dict:

      "segments": ["young_dhaka_student", "older_chittagong_pro", ...],
      "styles": ["joke_pun", "cultural_reference", ...],
      "matrix": [
        [0.65, 0.40, 0.30, ...],  // young_dhaka_student conversion rates
        [0.25, 0.55, 0.45, ...],  // older_chittagong_pro conversion rates
        ...
      ]

```

**Function:** `get_recommendations(brand_id, date_from, date_to)`

```python
def get_recommendations(brand_id: str, date_from: str, date_to: str) -> list:
    """
    Generates plain-language recommendations from the data.
    
    Logic:
    1. Find top 3 (segment, format/style) combos by conversion rate
    2. Find bottom 3 underperformers
    3. Identify any cultural moments approaching (next 7 days)
    4. Generate human-readable recommendations
    
    Output:
    [
      "Generate more GIF + joke combos for young urban users (47% conversion).",
      "Reduce stat-hook ads for under-25 segment (12% conversion).",
      "Eid is in 5 days — start cultural reference campaigns now."
    ]
    """
```

The recommendation generator can use either rule-based logic (for MVP) or call the LLM with the data summary (for richer output). LLM-based is more impressive in the demo — costs almost nothing.

### 8.4 Frontend rendering

Keep it simple:
- Use Recharts or Chart.js for the format winner bar chart
- Use a CSS grid with conditional background colors for the segment matrix (no fancy heatmap library needed)
- Recommendations are plain `<p>` tags with bullet points

### 8.5 Visual style

- Brand colors only where they don't conflict with chart readability
- Numbers should be large and readable from across the room (demo viewing distance)
- Use 1 accent color for "winning" (green) and 1 for "losing" (muted red/gray)

## 11. Data Seeding (Pre-Demo Preparation)

Before the demo, run a script to populate the events database with simulated data. This gives the dashboard something to display.

### 11.1 Seeding script

**File:** `/scripts/seed_demo_data.py`

```python
"""
Seeds the events database with simulated conversion data.
Run once before the demo.

Generates events for:
- 5 brands
- 5 personas
- ~10 unique context combinations
- ~5 ads per (brand, context) combination
= ~1250 ads × simulation = ~1250 events

This is enough to populate a meaningful dashboard with 
clear winners and losers per segment.
"""

def seed():
    brands = load_brand_kits()
    personas = load_personas()
    contexts = generate_context_grid()  # ~10 combos
    
    for brand in brands:
        for context in contexts:
            ads = load_pregenerated_ads(brand, context)  # ~5 per combo
            for persona in personas:
                for ad in ads:
                    converted = simulate_conversion(persona, ad, context)
                    log_conversion_event(
                        ad_id=ad["ad_id"],
                        persona_id=persona["id"],
                        brand_id=brand["brand_id"],
                        context=context,
                        ad_tags=ad["ad_tags"],
                        converted=converted,
                        source="simulated"
                    )
```

### 11.2 Time investment

Running the seed script takes a few seconds. Set this up on Day 2 once the format taxonomy and persona files are stable.

    method: "POST",

## 13. Privacy & Data Handling

### 13.1 What's stored

- Conversion events (yes/no per ad)
- Context snapshots (city, weather, time — no individual user identifiers)
- Persona snapshots (age bracket, occupation mode, NOT specific identity)
- Ad tags

### 13.2 What's never stored

- Individual user IDs (only persona IDs in MVP, or brand-anonymized IDs in production)
- Cross-site browsing behavior
- Sensitive inferred attributes
- Raw personal data

### 13.3 Production data handling note

In production:
- Brand owns the user. The system stores hashed/anonymized user IDs only.
- All conversion data tied to user is held by the brand; the system aggregates anonymously for pattern learning.
- GDPR/data protection: brands include opt-out in their existing privacy notices.

State this clearly in the pitch. Many MarTech tools cannot say "we don't store individual user identities" — you can.

## 17. What Judges Will See

When you click to the dashboard tab in your demo:

1. **Three seconds in:** Judge sees the bar chart and understands "GIFs win."
2. **Ten seconds in:** Judge sees the matrix and understands "different formats win for different people."
3. **Twenty seconds in:** Judge reads the recommendations and understands "the system tells the brand what to do next."

Total dashboard explanation time: **30 seconds.** Then you move on. Do not over-explain it.

### 17.1 What to say during the dashboard segment

> *"This is what the brand sees. We tag every generated ad by format and style. We track one signal — did the user take the offer. Over a week, patterns emerge. The brand doesn't read dashboards; they read recommendations."*

That sentence sequence is the entire dashboard pitch.
